Add a menu navigation breadcrumb bar and history system to pygame-menu. Currently, when navigating through nested submenus, users have no visual indicator of where they are in the menu hierarchy, and there's no programmatic way to inspect or manipulate the navigation history. Implement a BreadcrumbBar widget that renders a clickable breadcrumb trail (e.g., "Main Menu > Settings > Audio") at a configurable position (top or bottom of the menu). Each breadcrumb segment should be clickable to jump directly back to that ancestor menu. The separator character, font, colors (normal, hover, active), and padding should be configurable via new theme properties (breadcrumb_font, breadcrumb_font_color, breadcrumb_font_color_hover, breadcrumb_separator, breadcrumb_padding, breadcrumb_background_color). Also add a NavigationHistory class that tracks the menu navigation stack and provides history.back(), history.can_go_back(), history.get_path() (returns list of menu titles), and history.go_to(index) The breadcrumb bar should use the NavigationHistory internally. Hook the history tracking into Menu's _open() and _back() methods. Write tests for breadcrumb rendering, click-to-navigate behavior, history stack operations, and edge cases like circular menu references.

The 70/30 Execution Strategy:

Turn 1 (70%): Create pygame_menu/navigation.py with NavigationHistory class (stack operations, path tracking, go_to by index). Create the BreadcrumbBar class in the same module that renders the breadcrumb trail as a horizontal row of clickable text segments with separators. Add breadcrumb theme properties to themes.py. Hook history tracking into Menu._open() and Menu._back(). Add menu.get_navigation_history() and menu.set_breadcrumb_bar(enabled=True, position='top') APIs.

Turn 2 (30%): Implement click handling on breadcrumb segments (mouse hit detection on each segment's rect, triggering history.go_to()). Add hover color change on mouseover. Handle edge cases: breadcrumb trail too wide for menu (truncate with ellipsis from the left), history cleanup when menus are removed, breadcrumb updates when navigating via MenuLink widgets. Write comprehensive tests.

Why it fits the constraint:
navigation.py is ~160 lines (NavigationHistory ~60, BreadcrumbBar ~100 including rendering and click handling). Theme additions add ~25 lines. Menu integration (open/back hooks, public API) adds ~50 lines. Tests add ~120+ lines. Total: ~355-400 lines across 5 files. Difficulty is appropriate because the breadcrumb must track navigation correctly across arbitrarily nested menus, click detection requires coordinate math relative to the menu surface, and Turn 3 will address issues like breadcrumb not updating when navigating via keyboard-only (no mouse), or history getting out of sync when menus are opened programmatically rather than through normal navigation.
