## Phase 1 Analysis

### Scope Validation: ✅ PASS

This task is well-scoped. It addresses a single, concrete bug/feature: when MonkeyType traces two different classes that share the same name but come from different modules (e.g., `xml.dom.minidom.Element` vs `xml.etree.ElementTree.Element`), the generated stubs produce conflicting bare `from X import Element` / `from Y import Element` lines with no disambiguation. The fix is to detect these naming conflicts and automatically rename the imports using `as` aliases.

**Why it's good scope:**

- It's a single focused problem (not "add a dashboard") — resolve import naming conflicts in stub rendering
- It's not trivial — requires understanding the full `Stub` rendering pipeline, modifying the abstract `render()` interface across the entire class hierarchy, introducing a context-passing mechanism, and adding tests
- It touches 2 source files (`monkeytype/stubs.py`, `monkeytype/cli.py`) and 1 test file (`tests/test_stubs.py`)
- It has a clear "done" condition: conflicting imports get `as` aliases and the generated stubs use those aliases in annotations

**Key things the gold-standard solution does:**

1. Introduces a `RenderContext` dataclass in `stubs.py` that holds an `import_renames` mapping (`(module, name) -> renamed_name`)
2. Adds `resolve_import_conflicts()` method on `ImportBlockStub` — detects same-named symbols from different modules and generates renames using module initial-letter prefixes (e.g., `xdm_Element` for `xml.dom.minidom.Element`)
3. Adds a `_get_unused_name()` helper with numeric suffix fallback for uniqueness
4. Updates all `render()` methods throughout the `Stub` hierarchy (`Stub`, `ImportBlockStub`, `AttributeStub`, `FunctionStub`, `ClassStub`, `ModuleStub`) to accept `context: RenderContext` as a keyword argument
5. Updates `RenderAnnotation` and the standalone `render_annotation()`, `render_parameter()`, `render_signature()` functions to thread the context through
6. Updates CLI call-sites (`apply_stub_handler`, `get_diff`, `print_stub_handler`) to pass `RenderContext()`
7. Adds `test_multiple_imported_symbols_same_name()` test that verifies the end-to-end behavior using `xml.dom.minidom.Element` and `xml.etree.ElementTree.Element`

**Additional notes about the repo:**

- Linting/formatting: `tox.ini` runs `black`, `isort`, `flake8`, `mypy` only on the `monkeytype/` directory (not `tests/`)
- The test suite is pytest-based
- The codebase uses `ABCMeta` for the abstract `Stub` class, `dataclass` for new data containers, and `typing` generics throughout

### My Opinions on Prompt Strategy

The initial prompt should describe the problem scenario (two different classes, same name, different modules, both traced in type signatures) and ask for the conflict resolution feature. It should mention that the solution needs to rename conflicting imports using `as` aliases and that annotations in the stubs need to use the renamed names. It should NOT spell out the exact implementation approach (RenderContext, two-pass rendering, etc.) — that's for the model to figure out. It should ask for tests.

For follow-up turns, we'd likely need to address:

- Whether it touched `cli.py` call-sites
- Test coverage
- Black/isort compliance on `monkeytype/` directory
